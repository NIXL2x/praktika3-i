## 7. Контрольные вопросы

**1. Назовите пять мест в Linux, где вредонос может прописать автозапуск. Чем они отличаются по уровню привилегий?**

1. **Cron** (`crontab -l`) — может быть пользовательским или системным (root)
2. **Systemd** (`/etc/systemd/system/`, `~/.config/systemd/user/`) — системный (root) или пользовательский
3. **Bashrc** (`~/.bashrc`) — только для конкретного пользователя, при входе в shell
4. **rc.local** (`/etc/rc.local`) — системный, запускается при загрузке (root)
5. **ld.so.preload** (`/etc/ld.so.preload`) — загружается в каждый процесс (root, крайне опасно)

**Отличие по привилегиям:** системные (cron root, systemd системный, rc.local) требуют root и дают высокий уровень доступа. Пользовательские (cron пользователя, bashrc, user systemd) работают с правами пользователя, но всё равно опасны.

---

**2. Чем lsof отличается от ss? Когда использовать каждый?**

- **ss** — показывает сетевые соединения и слушающие порты. Быстрый, современный, замена netstat. Показывает только сетевую активность.
- **lsof** — показывает все открытые файлы, включая сокеты, pipes, устройства. Позволяет увидеть, что делает процесс в деталях.

**Когда использовать:**
- `ss` — для быстрой проверки открытых портов и соединений
- `lsof` — для глубокого анализа конкретного процесса (какие файлы открыты, откуда запущен, какие соединения держит)

---

**3. Что означает строка (deleted) в выводе lsof? Почему вредоносы используют этот трюк?**

`(deleted)` означает, что файл был удалён с диска, но процесс всё ещё держит его открытым. В Linux файл физически остаётся на диске, пока есть открытый дескриптор.

**Почему вредоносы используют:**
- Исполняемый файл удаляется после запуска — его сложнее найти через `ls` или `find`
- В файловой системе следа нет, но процесс работает
- Можно скрыть сам файл вредоноса, оставив только процесс

---

**4. Systemd-сервис в ~/.config/systemd/user/ — нужен ли root чтобы его создать? Почему это опасно?**

Root **не нужен** — любой пользователь может создать и включить сервис в своей домашней папке.

**Почему опасно:**
- Вредонос, запущенный от пользователя, может прописать себя в автозапуск без прав root
- Сервис будет запускаться при каждом входе пользователя
- Не требует sudo, администратор может не заметить
- Особенно опасно на серверах, где пользователи имеют доступ через SSH

---

**5. Вредонос переименовал себя в systemd-journald. Как отличить настоящий системный процесс от поддельного?**

1. **Посмотреть путь:** `ls -la /proc/PID/exe` — покажет реальный путь к исполняемому файлу. Настоящий `systemd-journald` живёт в `/lib/systemd/` или `/usr/lib/systemd/`.
2. **Проверить родителя:** `ps auxf` — настоящий системный процесс запущен от systemd (PID 1).
3. **Посмотреть открытые файлы:** `sudo lsof -p PID` — настоящий `systemd-journald` держит открытым `/var/log/journal/`.
4. **Проверить подпись пакета:** `dpkg -S /proc/PID/exe` — покажет, из какого пакета файл.
