# ПР №3. Права доступа Linux и управление пользователями

## 1. Пользователи и группы

Созданные пользователи и их роли:
| Пользователь | Группа | Роль |
|---|---|---|
| alice | developers | Администратор проекта |
| bob | developers | Разработчик |
| carol | auditors | Аудитор |

Поля /etc/passwd (разбор строки alice):
alice:x:1001:1003::/home/alice:/bin/bash
1. `alice` — имя пользователя
2. `x` — пароль хранится в `/etc/shadow`
3. `1001` — UID пользователя
4. `1003` — GID основной группы
5. пусто — комментарий (GECOS)
6. `/home/alice` — домашний каталог
7. `/bin/bash` — оболочка входа
### /etc/shadow

В файле `/etc/shadow` хранятся хэши паролей пользователей и параметры политики паролей.

Файл недоступен обычным пользователям, так как содержит зашифрованные пароли, что защищает систему от их извлечения и атак перебора.

## 2. Права доступа chmod/chown

Итоговые права /srv/project/code: `chmod 770`, владелец alice:developers
- bob не мог создавать файлы (нет `w` у группы)
- carol (не в группе): не мог войти в директорию, потому что у остальных пользователей нет прав доступа (r-x отсутствует для other).

## 3. ACL

Задача: дать carol доступ на чтение без изменения группы.
Команда: `setfacl -m u:carol:r-x /srv/project/code`

- Вывод getfacl после изменения:
file: srv/project/code
owner: alice
group: developers

user::rwx
user:carol:r-x
group::rwx
mask::rwx
other::---
- carol может просматривать каталог
- carol не может создавать файлы

### Вывод:
ACL позволяет выдавать доступ конкретному пользователю без изменения групповой модели прав.

## 5. PAM

### pam_unix.so
Модуль отвечает за проверку пароля пользователя через `/etc/shadow`.
### Значение required
Если модуль завершится ошибкой, аутентификация будет отклонена.
### Политика паролей
Файл `/etc/pam.d/common-password`: pam_unix.so obscure yescrypt
- `obscure` — базовая проверка сложности пароля
- `yescrypt` — алгоритм хэширования
### Минимальная длина пароля
Добавляется параметр:
- minlen=12

## 4. Контрольные вопросы

### 44. Что означает запись chmod 640 и кто при этом может читать файл?
`chmod 640` означает:
- владелец: `rw-` (чтение и запись)
- группа: `r--` (только чтение)
- остальные: `---` (нет доступа)
Читать файл могут только **владелец и пользователи группы**, к которой принадлежит файл.
---
### 45. Чем setuid-бит на исполняемом файле отличается от sudo? Приведите пример.
`setuid` — это специальный бит на исполняемом файле, при котором программа запускается с правами владельца файла (обычно root).
`sudo` — механизм, который позволяет выполнять команды с повышенными правами на основе правил в `/etc/sudoers`.
Разница:
- `setuid` встроен в файл программы
- `sudo` управляется политиками и требует авторизации пользователя
Пример:
- `passwd` использует `setuid` для изменения `/etc/shadow`
- `sudo apt update` выполняется по правилам sudoers
---
### 46. Почему хэши паролей хранятся в /etc/shadow, а не в /etc/passwd?
Файл `/etc/passwd` доступен всем пользователям системы, поэтому хранить там пароли небезопасно.
Хэши паролей перенесены в `/etc/shadow`, который доступен только root.
Это защищает систему от:
- кражи хэшей паролей
- атак перебора (brute force)
---
### 47. Что произойдёт если дать пользователю bob запись sudo bash? Почему это опасно?
Если разрешить:
bob ALL=(ALL) NOPASSWD: /bin/bash
то пользователь сможет запустить shell с правами root.
Это опасно, потому что через shell можно выполнить любые команды, что фактически даёт полный root-доступ и обходит ограничения sudo.
---
### 48. Разница между командами su и sudo
- `su` — переключение на другого пользователя (обычно root), требует пароль целевого пользователя
- `sudo` — выполнение одной команды с правами root по правилам `/etc/sudoers`
С точки зрения безопасности:
- `sudo` более безопасен (логирование, ограничение команд)
- `su` даёт полный сеанс другого пользователя
---
### 49. Как запретить конкретному пользователю использовать sudo, не удаляя его из группы sudo?
Можно запретить через `/etc/sudoers`, например:
username ALL=(ALL) NOPASSWD: /bin/false
Или удалить/закомментировать его правила в sudoers через `visudo`.
