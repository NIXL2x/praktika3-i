# ПР №7. AppArmor, Capabilities и Docker

## 1. Linux Capabilities

### Разбор getcap /usr/bin/ping
cap_net_raw=ep:
- **cap_net_raw** — разрешение использовать RAW-сокеты (ping отправляет ICMP-запросы)
- **e** (effective) — capability активна и работает
- **p** (permitted) — capability разрешена для использования

### CapPrm / CapEff / CapBnd — в чём разница

| Поле | Что это |
|------|---------|
| **CapPrm** | Что процесс вообще может использовать (максимальный набор) |
| **CapEff** | Что реально активно и применяется прямо сейчас |
| **CapBnd** | Жёсткая граница — выше этого процесс никогда не прыгнет |

**У текущего shell:**
CapPrm: 0000000000000000
CapEff: 0000000000000000
CapBnd: 000001ffffffffff

Shell не имеет никаких активных привилегий, но максимально может получить почти все.

### setcap — демонстрация

**До:**
python3 /tmp/test-port.py 80 → DENIED (Permission denied)
python3 /tmp/test-port.py 8080 → OK

**После setcap cap_net_bind_service=ep:**
python3 /tmp/test-port.py 80 → OK

**Почему лучше чем sudo:** sudo даёт полный root — процесс может всё. setcap даёт ровно одно право — привязаться к порту. Взломают скрипт — смогут только порты открывать, а не систему ломать.

### Флаги e, i, p

- **e** — работает сейчас
- **i** — дочерние процессы тоже получат
- **p** — вообще разрешено использовать

`ep` — работает и разрешено, но дети не наследуют.
`eip` — всё то же самое + передаётся детям.

## 2. AppArmor

### Количество профилей
- **Enforce:** 24
- **Complain:** 23

### Результаты pr07-reader

| Действие | Без профиля | complain | enforce |
|----------|-------------|----------|---------|
| Читать /tmp/pr07-allowed.txt | ✅ | ✅ | ✅ |
| Читать /etc/shadow | ❌ | ❌ | ❌ |
| Писать в /tmp/pr07-output.txt | ✅ | ✅ | ✅ |
| Писать в /etc/pr07-hack.txt | ❌ | ❌ | ❌ |

Блокировок AppArmor в логах нет — ограничения от обычных прав файлов (DAC), а не от профиля. Профиль просто не мешает тому, что и так запрещено.

### Разбор строки DENIED
Блокировок не было

- `operation` — что пытались сделать (открыть файл)
- `profile` — какой профиль сработал
- `name` — к чему обращались
- `denied_mask` — что запретили (чтение)

## 3. Docker — изоляция

| Ресурс | Хост | Контейнер |
|--------|------|-----------|
| Процессы | 317 шт | 1 шт |
| Сеть | lo, ens33, docker0 | только lo |
| /etc/shadow | свой | свой (не хост-овый) |
| Монтирование | разрешено | запрещено |

Контейнер видит только свою песочницу. Хост для него не существует.

### Capabilities: обычный vs --privileged

**Обычный:**
CapEff: 00000000a80425fb
→ cap_chown, cap_dac_override, cap_net_bind_service, ... (14 штук)
**--privileged:**
CapEff: 0000003fffffffff
→ почти всё, включая cap_sys_admin, cap_sys_module, cap_sys_rawio
**Чего нет у обычного:** всего, что позволяет лезть в ядро, устройства, другие процессы, менять сеть и систему.

**Почему --privileged опасен:** это фактически root на хосте. Взломали контейнер — взломали всю машину. Можно монтировать диски, грузить модули ядра, читать память других процессов.

### Запуск не от root
uid=1000 gid=1000
E: Could not open lock file /var/lib/dpkg/lock-frontend - Permission denied
Не root — не может ставить пакеты. Просто и эффективно. Даже если взломают — apt-get install не получится.

## 4. Итоговый nginx

bash
docker run -d \
--name pr07-nginx \
--cap-drop ALL \
--cap-add NET_BIND_SERVICE \
--cap-add CHOWN \
--cap-add DAC_OVERRIDE \
--cap-add SETGID \
--cap-add SETUID \
-p 8080:80 \
nginx:alpine

CapEff:00000000000004c3 → cap_chown, cap_dac_override, cap_setgid, cap_setuid, cap_net_bind_service
Почему именно они: nginx должен слушать 80-й порт (NET_BIND_SERVICE), работать с файлами (CHOWN, DAC_OVERRIDE) и переключать пользователей (SETGID, SETUID). Всё остальное отрезали — не нужно.
## 4. Эшелонированная защита

| Слой | Инструмент | Что ограничивает |
|---|---|---|
| DAC | chmod/chown | Кто что может читать/писать в файлах |
| Capabilities | --cap-drop ALL + cap-add | Какие системные вызовы доступны процессу |
| MAC | AppArmor | Что процессу вообще разрешено делать на уровне ядра |
| Изоляция | Docker namespaces | Свои процессы, сеть, ФС — контейнер не видит хост |

## Контрольные вопросы
67. Чем DAC отличается от MAC? Почему одного DAC недостаточно?

DAC — владелец решает, кто получит доступ. MAC — политику задаёт администратор, пользователь не может её обойти. Одного DAC мало, потому что если взломают процесс, запущенный от твоего пользователя — злоумышленник получит доступ ко всем твоим файлам. MAC работает поверх и не даст прочитать даже то, что владелец разрешил, если политика запрещает.

68. Что означает запись cap_net_bind_service=eip? Чем ep отличается от eip?

cap_net_bind_service=eip — разрешение на привязку к портам <1024, причём оно активное (e), разрешённое (p) и наследуемое детьми (i).
ep — то же самое, но без наследования. Дочерние процессы не получат это право.

69. В чём разница между complain и enforce в AppArmor? Зачем нужен complain?

Enforce — блокирует нарушения и пишет в лог. Complain — только пишет, но не блокирует. Complain нужен, чтобы обкатать новый профиль: посмотреть, что программа реально делает, добавить недостающие правила, и только потом включать enforce. Иначе можно случайно сломать работу сервиса.

70. Docker использует то же ядро что и хост — почему тогда контейнер изолирован?

Изоляция — это про namespaces: каждый контейнер видит только свои процессы (PID), свою сеть (NET), свою файловую систему (MNT). Плюс cgroups режут ресурсы, а capabilities и seccomp ограничивают системные вызовы. Контейнер думает, что он один, хотя ядро общее — он просто не видит соседей.

71. Злоумышленник нашёл RCE в nginx. Nginx в контейнере без --privileged, с --cap-drop ALL --cap-add NET_BIND_SERVICE, под непривилегированным пользователем, с AppArmor. Что может и не может сделать злоумышленник?

Может: читать/писать файлы внутри контейнера, которые доступны пользователю nginx, использовать сеть, выполнять команды внутри контейнера.

Не может: вылезти на хост, монтировать диски, менять сетевые настройки, ставить пакеты, читать хостовый /etc/shadow, трогать другие контейнеры, обойти AppArmor.

72. Почему --privileged — антипаттерн? Когда всё же оправдан?

Потому что это почти root на хосте. Весь смысл контейнеров — изоляция, а --privileged её ломает. Оправдан только когда без него никак: системные агенты мониторинга, работа с GPU, Docker-in-Docker в CI. В продакшене — почти никогда.

## Выводы
Попробовал на практике три механизма защиты: capabilities, AppArmor и Docker. Каждый решает свою задачу: capabilities режут права процессов, AppArmor контролирует доступ к файлам и системным вызовам, Docker изолирует окружение целиком. Вместе они дают эшелонированную защиту — если один слой пробьют, остальные всё равно держат. Главный вывод: принцип наименьших привилегий реально работает. Вместо "дать root и забыть" лучше потратить время на настройку ограничений. Особенно запомнилось, что --privileged — это зло, и в нормальных условиях его быть не должно.
