**Почему подпись перестала быть действительной после изменения файла:**  
Цифровая подпись создаётся на основе хеша (отпечатка) файла. При любом изменении файла хеш меняется, и проверка подписи завершается ошибкой. Это гарантирует целостность данных — любое изменение обнаруживается.

**Что именно проверяет `gpg --verify`:**  
Команда проверяет два аспекта:
1. **Целостность** — не изменился ли файл после подписания (сравнение хешей).
2. **Аутентичность** — действительно ли подпись принадлежит указанному владельцу ключа (через проверку сертификата публичным ключом).

**Зачем проверять подпись локально, если GitHub уже показывает Verified:**  
Локальная проверка независима от GitHub — она не требует подключения к интернету и доверия к серверам GitHub. Это позволяет убедиться в подлинности коммита даже в офлайн-среде или при работе с удалёнными репозиториями, не принадлежащими GitHub. Плюс, злоумышленник не может подделать вывод `git log --show-signature`, даже если взломает веб-интерфейс GitHub.

## 5. rm vs shred

| Операция | Что происходит с данными на диске | Можно восстановить? |
|----------|-----------------------------------|---------------------|
| `rm` | Удаляется только запись в inode (метаданные). Сами данные остаются на диске в неизменном виде до момента перезаписи другими файлами | Да, пока блоки не перезаписаны — специальными утилитами (testdisk, photorec) |
| `shred` | Содержимое файла многократно перезаписывается случайными данными (3 прохода), затем финальный проход нулями, после чего файл удаляется | Практически нет (при условии, что не используется SSD) |

**Что увидели при попытке восстановления после shred:**  
После `shred` данные на месте файла — случайные байты или нули. Исходная информация безвозвратно уничтожена.

**Сценарий, когда нужно затирать свободное место:**  
При передаче компьютера/диска другому человеку или продаже. Даже после удаления файлов через `rm` данные могут оставаться на диске. Затирание свободного места перезаписывает остатки удалённых файлов, предотвращая их восстановление.

## Контрольные вопросы

### 1. В чём разница между шифрованием и подписью? Можно ли одновременно зашифровать и подписать файл?

**Шифрование** — преобразование данных в нечитаемый вид для сохранения конфиденциальности. Расшифровать может только владелец приватного ключа.

**Подпись** — создание криптографического хеша, зашифрованного приватным ключом, для подтверждения авторства и целостности. Подпись не скрывает содержимое.

Да, можно одновременно зашифровать и подписать:
   bash
gpg --encrypt --sign --recipient EMAIL файл

### 2. Почему публичный ключ можно раздавать всем, а приватный нельзя никому?
Публичный ключ предназначен для шифрования и проверки подписи — даже зная его, нельзя расшифровать сообщение или подделать подпись.

Приватный ключ позволяет расшифровывать сообщения и создавать подписи. Если он попадёт к злоумышленнику, тот сможет:

Расшифровать все перехваченные сообщения

Подписываться чужим именем (подделка документов, коммитов)

Это основа асимметричной криптографии — разделение ролей.

### 3. Что произойдёт с зашифрованными данными, если потерять приватный ключ?
Данные будут потеряны безвозвратно. Расшифровать их невозможно — криптостойкость алгоритмов гарантирует, что без приватного ключа подобрать расшифровку за разумное время нереально (даже с суперкомпьютерами).

Рекомендация: хранить резервную копию приватного ключа в надёжном защищённом месте.

### 4. Можно ли подделать Verified коммит на GitHub, не имея приватного GPG-ключа владельца?
Нет. GitHub проверяет подпись коммита с помощью публичного ключа, загруженного пользователем. Без приватного ключа невозможно создать криптографически корректную подпись, которую GitHub примет как Verified. Это гарантирует, что коммит действительно сделан владельцем ключа.

### 5. Почему shred плохо работает на SSD? Что использовать вместо него?
Проблема: контроллер SSD использует wear leveling (выравнивание износа) — запись может физически попадать в другие ячейки, а старые данные оставаться в «теневых» областях. shred не может гарантированно перезаписать все копии данных.

Что использовать:

blkdiscard — TRIM-команда для всего диска

hdparm --security-erase — ATA Secure Erase (встроенная функция SSD)

Полное шифрование диска (LUKS) с самого начала

### 6. Коллега говорит: «я отформатировал диск перед продажей — всё безопасно». Он прав?
Нет, не прав. Обычное форматирование удаляет только служебную информацию (файловую систему), но данные остаются на диске. Их можно восстановить утилитами вроде testdisk или photorec.

Безопасный вариант:

Многократная перезапись (для HDD) — shred, dd if=/dev/urandom

ATA Secure Erase (для SSD)

Или просто продавать диск с включённым полным шифрованием (LUKS), не передавая пароль

### 7. Чем отличается --detach-sign от встроенной подписи (--sign)?
Параметр	Результат
--sign (или -s)	Подпись встраивается внутрь выходного файла (обычно в бинарный или ASCII-формат). Исходные данные + подпись в одном файле.
--detach-sign	Создаётся отдельный файл подписи (.asc). Исходный файл остаётся неизменным.
Отдельная подпись удобна, когда:

Нужно сохранить исходный файл в первоначальном виде

Передавать файл и подпись разными каналами

Подписывать исполняемые файлы, архивы, ISO-образы
