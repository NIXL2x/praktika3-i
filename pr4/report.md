# ПР №4. Аудит событий: journalctl и auditd
## 1. journalctl — системный журнал
### Что зафиксировал журнал о команде sudo cat /etc/shadow
мая 05 19:57:40 nixlpc sudo[4140]:     root : TTY=pts/1 ; PWD=/home/nixl ; USER=root ; COMMAND=/usr/bin/cat /etc/audit/rules.d/security-course.rules

**Разбор полей:**
1. nixlpc: имя хоста (компьютера).
2. sudo[4140]: имя процесса и его идентификатор (PID), сгенерировавшего запись.
3. nixl: имя пользователя, который инициировал выполнение команды через sudo.
4. TTY=pts/1: идентификатор терминала, в котором была запущена команда.
5. PWD=/home/nixl: текущая рабочая директория пользователя в момент запуска.
6. USER=root: пользователь, с правами которого выполняется команда (целевой пользователь).
7. COMMAND=/usr/bin/cat /etc/shadow: полный путь к исполняемому файлу и его аргументы.
### Ошибки в системе с последней загрузки
При выполнении команды journalctl -p err -b были обнаружены ошибки, связанные с драйверами графической подсистемы и инициализацией некоторых системных модулей (например, ACPI). Критических ошибок, влияющих на безопасность ядра, не выявлено.
### Статистика входов
 * **Успешных входов:** 0 (внешние SSH-подключения не зафиксированы).
 * **Неудачных попыток:** 0.
## 2. auditd — настройка правил
### Применённые правила (вывод auditctl -l)
```text
[sudo] пароль для nixl: 
-w /etc/passwd -p rwa -k auth-files
-w /etc/shadow -p rwa -k auth-files
-w /etc/group -p rwa -k auth-files
-w /etc/sudoers -p rwa -k priv-files
-w /etc/sudoers.d -p rwa -k priv-files
-w /etc/ssh/sshd_config -p rwa -k ssh-config
-a always,exit -F arch=b64 -S open,openat -F success=0 -F key=access-denied
-a always,exit -F arch=b64 -S unlink,unlinkat -F key=file-delete
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -F auid!=-1 -F key=priv-exec

```
### Разбор записи SYSCALL
| Поле | Значение | Что означает |
|---|---|---|
| **auid** | 1000 | Audit ID: идентификатор пользователя nixl, начавшего сессию. |
| **uid** | 1001 | User ID: идентификатор пользователя bob, запустившего команду. |
| **euid** | 1001 | Effective UID: эффективные права процесса (bob). |
| **comm** | cat | Имя команды (процесса). |
| **exe** | /usr/bin/cat | Полный путь к программе. |
## 3. Расследование инцидента
### Хронология действий bob
| Время | Действие | Результат | Источник (команда поиска) |
|---|---|---|---|
| 20:12:36 | cat /etc/shadow | Отказ | ausearch -k access-denied |
| 20:12:51 | ls /root | Отказ | ausearch -k access-denied |
| 20:13:05 | cat /etc/sudoers | Успех | ausearch -ua bob -i |
### Как auditd помог расследованию
Подсистема auditd зафиксировала попытки доступа к файлам, которые закончились ошибкой (Permission Denied). Даже если пользователь bob не оставил следов в обычных системных логах приложения, ядро Linux сохранило запись о системном вызове openat, что позволило восстановить хронологию.
### Чем auditd лучше journalctl для задач аудита безопасности
journalctl собирает сообщения от приложений и системных служб, которые можно "обмануть" или просто не отправлять туда логи. auditd работает на уровне ядра и перехватывает системные вызовы. Он позволяет отслеживать auid (Audit ID), который сохраняется неизменным при переключении пользователей, что критично для атрибуции действий.

## 7. Контрольные вопросы
### 65. Чем принципиально отличается auditd от journald? Почему нельзя обойтись только journald?
journald — это система сбора логов от приложений и сервисов (прикладной уровень). auditd — это система аудита системных вызовов ядра (низкий уровень). Нельзя обойтись только journald, так как он не фиксирует события, о которых приложение "решило промолчать", а также не умеет отслеживать доступ к файлам на уровне ядра.
### 66.
Что такое auid и почему оно важнее uid при расследовании инцидентов с sudo?
auid (Audit User ID) — это идентификатор пользователя, который изначально вошел в систему. В отличие от uid, он не меняется при использовании команд su или sudo. Это позволяет ИБ-специалисту понять, какой именно человек совершил действие, даже если он стал root.
### 67. Что означает флаг -k в правиле auditd? Для чего он используется?
Флаг -k  задает произвольную текстовую метку  для правила. Он используется для быстрой фильтрации событий при поиске через ausearch -k <метка>, чтобы не просеивать тысячи нерелевантных записей.
### 68. Чем -w отличается от -a always,exit в правилах auditd? Когда использовать каждый?
-w используется для наблюдения за конкретными файлами или директориями. -a always,exit  используется для более сложной логики: фильтрация по номеру системного вызова, по результату выполнения (успех/ошибка) или по архитектуре процессора.
### 69. Злоумышленник удалил файл /var/log/audit/audit.log. Как это помогает ему скрыть следы и как от этого защититься?
Удаление файла скрывает локальную историю действий. Защититься можно настройкой удаленного логирования (отправка логов на SIEM или выделенный сервер логов в реальном времени через audispd), а также ограничением прав на директорию /var/log/audit/.
### 70. Что нужно сделать чтобы правила auditd сохранились после перезагрузки?
Правила должны быть записаны в файлы с расширением .rules в директории /etc/audit/rules.d/. При загрузке служба auditd выполняет команду augenrules, которая объединяет эти файлы в итоговый список правил.
